1. What this document explains

Intelligence reports show a TREND ANALYSIS (NDVI/EVI/RVI) chart with two visual layers: smoothed lines and scatter markers labeled “(raw)”. This guide explains, step by step, how those values are produced in code, what “raw” really means in this project, and why you can still see optical (NDVI/EVI) points during the Indian monsoon (June–July and beyond).

Key takeaway: “Raw” in our reports means pre-Whittaker smoothing in Python — not unprocessed satellite bands and not necessarily cloud-free parcel observations.

2. End-to-end pipeline (fetch → chart → PDF)

5. Sentinel-2: cloud mask and index formulas (NDVI, EVI)

We use the harmonized surface reflectance collection COPERNICUS/S2_SR_HARMONIZED. Cloudy pixels are masked using the Scene Classification Layer (SCL), then reflectance is scaled to 0–1.

5.1 Cloud masking (pixel level)

SCL classes removed: 3 (cloud shadows), 8 (medium cloud), 9 (high cloud), 10 (thin cirrus).

def maskS2clouds(image):
    scl = image.select('SCL')
    mask = (
        scl.neq(3).And(scl.neq(8))
        .And(scl.neq(9)).And(scl.neq(10))
    )
    return image.updateMask(mask).divide(10000).copyProperties(image, ["system:time_start"])

5.2 NDVI and EVI (computed on each image in GEE)

IndexBandsFormula in code
NDVIB8 (NIR), B4 (RED)normalizedDifference(['B8','B4']) → (NIR−RED)/(NIR+RED)
EVIB8, B4, B2 (BLUE)2.5 * (NIR−RED) / (NIR + 6*RED − 7.5*BLUE + 1)
def compute_indices(img):
    ndvi = img.normalizedDifference(['B8','B4']).rename('NDVI')
    evi = img.expression(
        '(2.5*((NIR-RED)/(NIR+6*RED-7.5*BLUE+1)))',
        {'NIR': img.select('B8'), 'RED': img.select('B4'), 'BLUE': img.select('B2')}
    ).rename('EVI')
    return img.addBands([ndvi, evi]).select(['NDVI','EVI'])

5.3 Scene-level cloud filter + full-scene fetch

  • Scene filter: entire tile must have CLOUDY_PIXEL_PERCENTAGE < 20 (metadata for the granule, not your parcel alone).
  • Current behavior: all Sentinel-2 scenes that pass the filters are fetched. The old limit_to_60() thinning call is intentionally commented out.
s2_col = (ee.ImageCollection('COPERNICUS/S2_SR_HARMONIZED')
    .filterDate(start_str, end_str)
    .filterBounds(polygon)
    .filter(ee.Filter.lt('CLOUDY_PIXEL_PERCENTAGE', 20))
    .map(maskS2clouds)
    .map(compute_indices))
# s2_col = limit_to_60(s2_col)

6. Sentinel-1: RVI (radar — works through clouds)

Radar is not blocked by monsoon clouds. RVI is computed from VV and VH in decibels, converted to linear power first. 

def compute_rvi(img):
    vv = ee.Image(10).pow(img.select('VV').divide(10))
    vh = ee.Image(10).pow(img.select('VH').divide(10))
    rvi = vh.multiply(4).divide(vv.add(vh)).rename('RVI')
    return img.addBands(rvi).select(['RVI'])

s1_col = (ee.ImageCollection('COPERNICUS/S1_GRD')
    .filterDate(start_str, end_str)
    .filterBounds(polygon)
    .filter(ee.Filter.eq('instrumentMode','IW'))
    .filter(ee.Filter.listContains('transmitterReceiverPolarisation','VV'))
    .filter(ee.Filter.listContains('transmitterReceiverPolarisation','VH'))
    .filter(ee.Filter.eq('orbitProperties_pass','DESCENDING'))
    .map(compute_rvi))

Note: Sentinel-1 was already not passed through limit_to_60. Sentinel-2 and Dynamic World are now also fetched without that thinning step.

7. What extract_timeseries means

This is a local helper function (not a GEE API name). It converts an ImageCollection into a pandas table: one row per satellite acquisition date, one column per index/band.

For each image in the collection it:

  1. Computes the mean of each band inside the parcel polygon at 10 m scale.
  2. Stores the image acquisition date as YYYY-MM-dd.
  3. Downloads all rows from GEE with getInfo() and builds a sorted DataFrame.
def extract_timeseries(collection, bands):
    def reducer(img):
        stats = img.reduceRegion(
            reducer=ee.Reducer.mean(),
            geometry=polygon,
            scale=10,
            bestEffort=True,
            maxPixels=1e13
        )
        props = {'date': img.date().format('YYYY-MM-dd')}
        for b in bands:
            props[b] = stats.get(b)
        return ee.Feature(None, props)
    fc = collection.map(reducer).filter(ee.Filter.notNull(bands))
    info = fc.getInfo().get('features', [])
    rows = [f['properties'] for f in info]
    df = pd.DataFrame(rows)
    df['date'] = pd.to_datetime(df['date'])
    return df.sort_values('date')

df_s2 = extract_timeseries(s2_col, ['NDVI','EVI'])
df_s1 = extract_timeseries(s1_col, ['RVI'])
Important: bestEffort=True allows a mean even when only part of the parcel has valid pixels after cloud masking. There is no rule requiring e.g. 50% of the parcel to be cloud-free. This matters heavily during monsoon.

11. Why monsoon (June–July) still shows NDVI/EVI points

Over much of India, the southwest monsoon (roughly June–September) reduces usable optical imagery. Many users expect no NDVI points in July. Our pipeline can still show some — for technical reasons below. This is usually sparse real observations, not invented dates, but confidence is lower.

11.1 Scene cloud % is for the whole tile, not your field

CLOUDY_PIXEL_PERCENTAGE < 20 applies to the entire Sentinel-2 granule. A scene can pass while your parcel still has patchy clouds — or a relatively clear field can contribute to a scene that passes the 20% rule.

11.2 SCL masking is imperfect + partial means

After SCL removes cloud classes, remaining pixels are averaged. With bestEffort=True, GEE may return a mean from only the clear fraction of the parcel. Monsoon points often mean: “we got some masked NDVI,” not “full clear-sky view.”

11.3 Monsoon ≠ zero satellite passes

Sentinel-2 revisits every few days. Even in cloudy months, a fraction of scenes pass filters. Because the current code no longer applies limit_to_60, all filtered June/July optical scenes are eligible to appear as raw markers.

11.4 Do not confuse NDVI with RVI in monsoon

RVI (orange markers) comes from Sentinel-1 radar and is largely unaffected by clouds. NDVI/EVI (green/blue markers) are optical. Monsoon charts often look “fuller” because of SAR — check the legend.

11.5 Smoothed lines can hide data gaps

A continuous green line through July does not mean NDVI was measured every day. Whittaker smoothing bridges gaps between sparse scene dates. Always compare marker density to the line.

12. Does this create false data or reduce accuracy?

QuestionAnswer
Are scatter dates fabricated?No — each marker ties to a real filtered GEE scene date.
Are daily smoothed values measured?Not all — resample + Whittaker synthesize values on days without observations.
Are monsoon optical points always wrong?No — but they are often noisier and should be lower confidence.
Impact on crop verdict?Predictions use smoothed daily series — monsoon gaps can be under-filled or over-smoothed, affecting activity %.

14. Possible improvements (future work)

  • Require minimum % of parcel pixels valid after cloud mask before keeping a scene.
  • Tighten CLOUDY_PIXEL_PERCENTAGE (e.g. < 10) or add parcel-level QA.
  • Plot optical and radar on separate panels; annotate scene count per month on the PDF.
  • Store true scene-level series in pickle (vegetation_indices currently saves smoothed daily data).
  • Flag annexure rows as observed vs interpolated/smoothed.
